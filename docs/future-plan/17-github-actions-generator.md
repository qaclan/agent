# 17 - GitHub Actions Generator

Deep-dive on feature #17 from [feature-ideas.md](../feature-ideas.md). Covers what makes a good generated workflow vs a generic template, and what QAClan-specific context we can bake in.

---

## The problem

Getting QAClan into CI requires a developer to:

1. Know that `.github/workflows/` is where to put the file.
2. Know the GitHub Actions YAML syntax.
3. Know how to install QAClan in a CI runner.
4. Know which suite to run, in which environment, on which browsers.
5. Know how to handle secrets in GitHub Actions (`${{ secrets.X }}`).
6. Know how to upload artifacts (screenshots, traces).

Most QAClan users are not CI experts. Even those who are spend 30+ minutes configuring a fresh workflow per project.

## Example

Generated workflow for a project with suite "smoke", environment "staging", and one secret `LOGIN_PASSWORD`:

```yaml
# Generated by QAClan — edit freely
name: QAClan E2E Tests

on:
  push:
    branches: [main]
  pull_request:

jobs:
  qaclan:
    runs-on: ubuntu-latest
    timeout-minutes: 30

    steps:
      - uses: actions/checkout@v4

      - name: Install QAClan
        run: curl -sSL https://qaclan.com/install.sh | bash
        env:
          QACLAN_INSTALL_SILENT: 1

      - name: Setup runtime
        run: qaclan setup --runtime-only

      - name: Run suite
        run: qaclan suite run smoke --env staging --reporter junit
        env:
          LOGIN_PASSWORD: ${{ secrets.LOGIN_PASSWORD }}
          # Add more env vars from your QAClan environment config here

      - name: Upload test results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: qaclan-results
          path: ~/.qaclan/runtime/runs/

      - name: Upload screenshots
        if: failure()
        uses: actions/upload-artifact@v4
        with:
          name: screenshots
          path: ~/.qaclan/screenshots/
```

User gets this in one click. No YAML knowledge needed.

## Impact

| Dimension | Effect |
|---|---|
| Time to first CI run | 30+ minutes of YAML writing → 2 minutes (download file, push, done). High. |
| Error rate | YAML is notoriously finicky (indentation, quoting). Generated files are syntax-valid. |
| Suite discoverability | Generator shows all available suites — user picks, no memorization. |
| Secrets awareness | Generator detects which env vars in the selected environment are marked secret → shows GitHub secrets instruction automatically. |
| Product adoption | CI integration = permanent visibility. Every CI run = QAClan usage. Users who see QAClan in CI don't abandon it. |

Overall: **high impact, very low implementation cost**. It's mostly template rendering.

## How this differs from existing tools

| Tool | What it does | Gap |
|---|---|---|
| **Playwright init** | `npm init playwright` asks "Add GitHub Actions?" → generates a basic Playwright CI workflow. | Generic. No QAClan. User still needs to install QAClan, configure suites, env vars manually. |
| **Cypress** | `cypress open` → guides through CI setup with a generic snippet. | Generic CI snippet. Not project-aware. |
| **GitHub's workflow templates** | Generic templates at repo creation. | Not tool-specific. |

**Our diff:** The generated file is **pre-filled** with:
- The user's actual suite name (picked from QAClan's suite list).
- The user's environment (from QAClan env table).
- Secret env vars (detected from env table rows where `is_secret = 1`).
- The correct QAClan install command for the CI platform.
- Upload artifacts for the actual paths QAClan uses.

It is not a generic template. It is a file that would work immediately if pasted into the repo.

## Generator inputs

The UI asks for:

| Field | Source | Example |
|---|---|---|
| Suite to run | Dropdown from `suites` table | "smoke" |
| Environment | Dropdown from `environments` table | "staging" |
| Trigger | Checkbox: push, PR, both | both |
| Runner OS | Dropdown: ubuntu, macos, windows | ubuntu-latest |
| Reporter format | Dropdown: junit, json, none | junit |

## Secret detection

For the selected environment, query env vars with `is_secret = 1`. For each:

```yaml
# Generated: for each secret env var named ADMIN_PASS
          ADMIN_PASS: ${{ secrets.ADMIN_PASS }}
```

And add a notice block in the output:

```
⚠️  Secrets detected. Before running this workflow, add the following to your
GitHub repo → Settings → Secrets → Actions:
  • ADMIN_PASS    (from your "staging" environment)
```

## CLI reporter flag

Generator requires `qaclan suite run <name> --reporter junit` to exist. If it doesn't exist yet, this feature should be paired with feature #18 (JUnit/Allure Export). Add the CLI flag first, reference it in the generator.

If JUnit isn't ready, use `--reporter json` or omit reporter and just rely on exit code for CI pass/fail.

## Output options

| Option | UX |
|---|---|
| Copy to clipboard | "Copied!" button. User pastes into repo manually. |
| Download file | Browser downloads `qaclan.yml`. User adds to `.github/workflows/`. |
| Direct PR (stretch goal) | GitHub API: fork or commit to branch, open PR with the file. Requires GitHub token from user. |

v1: copy + download. PR option is a stretch goal for v2.

## Implementation path

### Phase 1 — template + output

- New Flask route: `GET /api/github-actions/generate?suite_id=&env_id=&os=&triggers=`.
- Python Jinja2 template: `web/templates/github-actions.yml.j2`.
- Returns rendered YAML as text.
- UI: "CI Integration" page (or button on suite detail page). Shows dropdowns + preview of the generated YAML. Copy + Download buttons.

### Phase 2 — secret instructions

- Detect `is_secret = 1` env vars for selected environment.
- Show instruction panel: "Add these secrets to GitHub before running."
- Include deep link to GitHub secrets settings for the repo (if user provides repo URL).

### Phase 3 — other CI platforms

- Same template system extended to: GitLab CI (`.gitlab-ci.yml`), CircleCI (`.circleci/config.yml`), Jenkins (`Jenkinsfile`).
- Platform picker added to the CI Integration UI.

## Open questions

- **Install URL.** `curl -sSL https://qaclan.com/install.sh` — this URL must be stable and point to the latest binary. Coordinate with release infrastructure.
- **Pinned QAClan version.** Should the generated workflow pin a QAClan version (`qaclan@1.2.3`) or always use latest? Latest is simpler but can break workflows on upgrades. Add an optional version pin field in v2.
- **`qaclan suite run` CLI.** Does this command exist? Check [cli/commands/](../../cli/commands/) — if not, the generator must be paired with implementing this CLI path. The web runner exists; CLI runner may not.

## Next concrete step

Check whether `qaclan suite run <name>` CLI command exists. If yes, build the Jinja2 template + Flask route + minimal UI page immediately — it is almost pure templating. If not, track that as a dependency and implement the CLI command first.
